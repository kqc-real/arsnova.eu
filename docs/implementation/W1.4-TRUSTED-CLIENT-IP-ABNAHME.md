# W1.4-Abnahme: vertrauenswürdige Client-IP

**Stand:** 2026-07-24  
**Owner:** Backend / Security  
**Ticket:** PR-Slice W1.4  
**Abhängigkeit:** dokumentierter Einzel-Nginx mit `TRUST_PROXY_HOPS=1`

## Schutzmodell

- `resolveClientIp()` verwendet ausschließlich Express' bereits gemäß `trust proxy`
  berechnetes `req.ip`.
- Frei sendbare `CF-Connecting-IP`-, `True-Client-IP`-, `X-Forwarded-For`- und
  `X-Real-IP`-Header werden nie direkt gelesen.
- Ohne Express-IP-Kontext wird nur die direkte Socket-Adresse verwendet; fehlt auch
  diese, landet die Anfrage im stabilen unbekannten Bucket `0.0.0.0`.
- Session-Create, Session-Code-Lockout, MOTD, Public Creates und 429-Telemetrie
  nutzen denselben Resolver.
- Nginx überschreibt `X-Forwarded-For` mit `$remote_addr`; das Backend vertraut
  genau einem Proxy-Hop. Ein zusätzliches CDN/WAF erfordert ein neues
  Trust-Proxy-Review.

## Regressionen

- Unterschiedliche gefälschte CF-/True-Client-/XFF-/X-Real-IP-Header ändern bei
  gleichem `req.ip` weder Resolver-Ergebnis noch Rate-Limit-Bucket.
- Ohne Express-Kontext werden Proxy-Header ignoriert und die Socket-Adresse genutzt.
- Bestehende Public-Create-, Session-Create-, Session-Code- und MOTD-Tests bleiben
  grün.

## Slice-Grenze

W1.4 ändert keine Schwellen. Der noch IP-basierte Session-Code-Lockout ist für
große Hörsaal-NATs ungeeignet und wird unmittelbar in W1.5 durch client-/codebezogene
Budgets mit globalem Soft-Cap ersetzt. Gültige Joins dürfen dort nicht am globalen
Fehlversuchsbudget scheitern.
